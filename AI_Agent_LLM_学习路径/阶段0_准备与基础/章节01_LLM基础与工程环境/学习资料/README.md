# 章节 01：LLM 基础与工程环境

## 学习目标

- 理解大模型的最小知识框架：预训练、指令微调、对齐、上下文窗口、采样、工具调用。
- 区分 Base Model、Instruct Model、Chat Model、Reasoning Model 的实际差异。
- 搭建自己的实验环境，为后续所有章节做准备。

## 你必须掌握的理论

### 1. LLM 工作流

- 预训练：模型通过大规模语料学习统计规律。
- 指令微调：让模型更像一个“听得懂任务”的助手。
- 对齐：通过偏好数据、规则或反馈，让模型回答更可控。
- 推理：用户输入经过 tokenizer 编码，模型逐 token 生成输出。

### 2. 关键概念

- Token：模型处理文本的基本单位，不等于字、不等于词。
- Context Window：模型一次性能看到的 token 上限。
- Temperature：控制输出随机性。
- Top-p：控制采样概率质量。
- Hallucination：模型在缺少依据时仍生成貌似正确的内容。
- Tool Calling：模型不直接回答，而是规划并调用工具获取信息。

### 3. 为什么只会 Prompt 不够

- Prompt 可以提升基础表现，但无法替代检索、工具、评测、监控。
- 真正可用的系统需要：
  - 稳定输入输出结构
  - 错误处理
  - 可追踪日志
  - 可重复评测
  - 成本和延迟控制

## 你需要搭建的环境

- Python 3.11+
- 虚拟环境工具：`venv` 或 `conda`
- Git
- Docker
- VS Code 或其他 IDE
- 一个可用的 LLM API 提供方

## 本章建议实操

1. 建一个最小 Python 项目。
2. 用环境变量保存 API Key。
3. 写一个最简单的 `chat.py`，把用户输入发给模型并打印结果。
4. 记录三组不同参数下的回答差异：`temperature=0`、`0.7`、`1.0`。

## 常见误区

- 误区 1：上下文窗口越大越好。
  实际上窗口越大，成本和延迟通常也越高，注意力分配也未必最优。

- 误区 2：模型参数量越大，一定越适合业务。
  实际上还要看任务类型、延迟预算、成本预算和部署环境。

- 误区 3：模型回答得像真的，就说明系统可靠。
  真正可靠必须能验证来源、流程和输出约束。
